The Standard Definition of NIL

Introduction

This document provides the formal definition of "NIL", an algorithmic language for computer programming.

NIL is a recursive acronym which stands for "NIL isn't Lisp".

The purpose fo this document is to promote the portability of NIL programs across a variety of implementations and hardware.

Scope

This document specifies the format, meaning, and validation of NIL programs.

This document does not specify requirements for the host system, hardware, or the internal mechanisms of an implementation.

Conformance

This document in its entirety constitutes a set of requirements for an implementation

An implementation of the NIL programming language is said to be a "conforming implementation" if it fulfills the requirements as set out in this document.

Requirements which are beyond the scope of this document are explicitly classified as "implemntation dependent".

Requirements which must be present in some implementation dependent capacity are explicitly classified as "implementation defined".

Principals of Design

Be useful.

Sessions

Sessions are the abstract interface through which a programmer may utilize NIL.

A Session is minimally a live process which exposes a duplex stream of UTF-8 encoded Unicode® characters and maintains a Module. The programmer will submit written Programs via this stream which may affect the bindings in the Sessions's Module, and the Session will return the results of the execution as the written Program which constructs it.

A Session will always evaluate a Program submitted by the programmer so long as it can be validated.

A Session will provide capabilities for transforming NIL Programs into programs which are understood by the host.

A Session will provide capabilities for inspecting NIL Programs, and the effects of their execution.

Context

The Module which is maintained by an active Session is referred to as the Context.

When a Session begins evaluation of a Program, it shall use the Context to determine the meaning and effects of the program.

The Context contains sub-Modules which are accessible to Programs. They are...

  • Default
  • System
  • Auxiliary

The Context implicitly exposes these Modules to Programs as if by evaluating the following Program...

(define Default (load 'Default)
System (load 'System)
Auxiliary (load 'Auxiliary))
(use Default System Auxiliary)
