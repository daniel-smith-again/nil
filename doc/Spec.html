                                ╭──────────────╮
                                ╽ Introduction │
╭──────────────────────────────────────────────╯
├ This document provides the formal definition of "NIL", an algorithmic language
│ for computer programming. 
│
├ NIL is a recursive acronym which stands for "Nil Isn't Lisp".
│
├ The purpose of this document is to promote the portability of NIL programs
│ across a variety of implementations and hardware.
│
│                                  ╭───────╮
│                                  ╽ Scope │
│╭─────────────────────────────────────────╯
│├ This document specifies the format, meaning, and validation of NIL programs.
││
│├ This document does not specify requirements for the host system, hardware,
││ or the the internal mechanisms of an implementation.
│╰╼
│                               ╭─────────────╮
│                               ╽ Conformance │
│╭────────────────────────────────────────────╯
│├ This document in its entirety constitutes a set of requirements for an
││ implementation.
││
│├ An implementation of the NIL programming language is said to be a
││ "conforming implementation" if it fulfills the requirements as set out in
││ this document.
││
│├ Requirements which are beyond the scope of this document are explicitly
││ classified as "implementation dependent".
││
│├ Requirements which must be present in some implementation dependent capacity
││ are explicitly classified as "implementation defined".
│╰╼
│                               ╭─────────────╮
│                               ╽ Conventions │
│╭────────────────────────────────────────────╯
│├ Chapters, Sections, and Subsections are denoted by nesting of runners down
││ the left margin.
││
│├ Macros, Default Functions, and other operations provided by NIL are 
││ documented by "card" figures which give the abstract name of the operation, 
││ the form of its invocation, and the type constraints which evaluation should 
││ conform to. An example card is given below.

│                           ╭──────────────────────╮
│                           ╽ Principals of Design │
│╭─────────────────────────────────────────────────╯
│├ There exists a notion of "small" and "large" programming. This refers to the
││ disparity between abstractions over "small" single-thread processes, compute
││ kernels, and constrained environments, and "large" multi-process systems, high 
││ complexity software with many components, and homogeneous computing 
││ architectures. As software continues to grow in complexity, the gulf between 
││ what is "small" and what is "large" increases. Additionally, the domains of 
││ "small" and "large" grow orthogonally in terms of what is needed to express
││ programs in each. NIL is designed to bridge that gap.
││
││ Both "large" and "small" programs require a certain level of expressivity.
││ However, these domains are not mutually exclusive. They require exactly the 
││ same form of expressivity. To this end, NIL provides a macro 
││ system which allows the structure and meaning of programs to be transformed 
││ programmatically in order for a domain to be modelled in its own terms while 
││ maintaining a close correspondence with the domain of the host.
││
││ Increasingly so, programs are transmitted and recieved between disparate 
││ hosts with varying levels of trust and secure capability. It becomes necessary
││ to validate programs in their initial state without allowing execution. There
││ are known limitations to validation such as detecting divergence in
││ recursion, however these limitations tend to be of little practical 
││ significance. Much more often, discrete properties of programs are the 
││ subject of validation such as whether or not a program will corrupt data, 
││ access secure resources without permission, or simply behave according to how
││ it appears. NIL provides a notion of validation such that if a program is 
││ invalidated, it can be safely rejected as malformed. This allows a program, 
││ as it is stored for transmission, to contain embedded constraints pertaining 
││ to its behavior and effects.
││
││ Both "large" and "small" programs require some degree of encapsulation. For
││ "small" programs, encapsulation serves to hide differences in hosts which have
││ no effect on the successful execution of a program. For "large" programs, 
││ encapsulation allows multiple self-contained, monolithic components to 
││ interface with each other. NIL provides first-order modules as the primary 
││ means of encapsulation. Modules may encapsulate both state and programs which 
││ allow them to fulfill a variety of approaches to program organization and 
││ the design of interfaces.
││
││ All computer programs, in both "large" and "small" domains, face challenges
││ such as the proper handling of resources, multi-process communication, and 
││ the handling of internal and external conditions such as errors. In all these
││ cases, some element of external state manifests. NIL provides a system of 
││ typed effects to address this. Typed effects allow for the introduction of 
││ non-local control flow in a rigorous fashion. Programs may employ effects to
││ abstract over, and react to, external or otherwise asynchronous stateful 
││ changes in a predictable manner.
││
││ NIL also outlines a novel interface for the creation and execution of 
││ programs. This interface is called "session based programming". This is in 
││ contrast to the practices of "batch compilation" or "dynamic programming". To
││ illustrate the difference, imagine a painter who is beginning a new 
││ painting... 
││ 
││ If the painter employed "batch compilation" as a method, they would be 
││ required to finish the painting as fast as possible in a single attempt, with
││ no mistakes. If the painter is not able to finish, they must burn the 
││ incomplete painting, dispose of all their paints, and begin anew with their 
││ next attempt.
││
││ If the painter employed "dynamic programming" as a method, they would never 
││ be able to complete the painting. The paint would never dry, and each brush 
││ stroke would affix that brush to the canvas. Not only this, the painting 
││ would be permanently stuck to the easel, and the tubes of paint and paint 
││ thinner would need to be stored with the painting at all times.
││
││ It's apparent from these analogies that neither of the existing approaches 
││ is feasible. It's possible to paint in that manner, but extremely tedious.
││ Session based programming provides a compromise between the two extremes.
││ The painter can paint over multiple periods without needing to destroy their 
││ incomplete work. At the same time the painter may discard their paints, 
││ brushes and easel when they're finished. In terms of programming, a 
││ programmer may use NIL to run and inspect partial or incomplete programs 
││ within a managed context. When the programmer decides their program is 
││ complete, they may extract a static executable from that context.
│╰╼ 
╰╼

                                  ╭──────────╮
                                  ╽ Sessions │
╭────────────────────────────────────────────╯
├ Sessions are the abstract interface through which a programmer may utilize
│ NIL.
│
├ A Session is minimally a live process which exposes a duplex stream of UTF-8
│ encoded Unicode® characters and maintains a Module. The programmer will 
│ submit written Programs via this stream which may affect the bindings in the
│ Session's Module, and the Session will return the results of execution as the 
│ written Program which constructs it.
│
├ A Session will always evaluate a Program submitted by the programmer so long 
│ as it can be validated.
│
├ A Session will provide capabilities for transforming NIL Programs into 
│ programs which are understood by the host.
│
├ A Session will provide capabilities for inspecting NIL Programs, and the 
│ effects of their execution.
│
│                                 ╭─────────╮
│                                 ╽ Context │
│╭──────────────────────────────────────────╯
│├ The Module which is maintained by an active Session is referred to as the
││ Context.
││
│├ When a Session begins evaluation of a Program, it shall use the Context to 
││ determine the meaning and effects of the program.
││
│├ The Context contains sub-Modules which are accessible to Programs.
││ They are...
││ ⎡
││ ⎢ ⋅ Default
││ ⎢ ⋅ System
││ ⎢ ⋅ Auxiliary
││ ⎣
││ 
│├ The Context implicitly exposes these Modules to Programs as if by evaluating 
││ the following Program...
││
││                  ⎛                                      ⎞
││                  ⎜ (define Default   (load 'Default))   ⎟
││                  ⎜ (define System    (load 'System))    ⎟
││                  ⎜ (define Auxiliary (load 'Auxiliary)) ⎟
││                  ⎜ (use Default System Auxiliary)       ⎟
││                  ⎝                                      ⎠
││
│├ The Default Module shall contain the definitions which form the NIL language,
││ or in other words the definitions set out in this document. A complete 
││ listing is given here.
││
│├ The System Module shall contain implementation defined definitions which 
││ facilitate access to the host system.
││
│├ The Auxiliary Module shall contain any other implementation defined 
││ definitions which are not associated with the System Module and are outside 
││ the scope of this document.
││
│├ The Session may only ever refer to a single Module when evaluating a Program.
││ The Module referred to by the Session is called the Top-Level.
││ When a Session is first started, the Top-Level will always be the Context, 
││ the implicit Module which uses the Default, System, and Auxiliary Modules.
││ The Top-Level may be set programmatically to a different Module.
│╰╼
│
│                              ╭────────────────╮
│                              ╽ Session Macros │
│╭──────────────────────────────────────────────╯
│├ NIL provides Macros which allow the duplex stream to be accessed 
││ programmatically. They are defined here as Macros since their inner 
││ mechanisms may be implementation defined and may result in different behavior
││ depending on the host system. They are as follows...
││
││ ╭──────╮  
││ │ Load ╰─────────────────────────────────────────────────────────────────╮
││ │ (syntax Load (load) (load path) (((Path) → Load-Failed Module) path)   │
││ ├┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┤
││ │ Load will accept a path list that points to location of a saved NIL    │
││ │ Module.                                                                │
││ │                                                                        │
││ │ Load will convert the saved Module file into a live Module instance    │
││ │ which will be returned as the result.                                  │
││ │                                                                        │
││ │ If a path list is given that does not point to the location of a NIL   │
││ │ Module, this Expression will call the 'load-failed effect.             │
││ │                                                                        │
││ │ If there is an implementation dependent failure when the Session       │
││ │ attempts to load the Module, this Expression will call the             │
││ │ 'load-failed effect.                                                   │
││ ╰────────────────────────────────────────────────────────────────────────╯
││
││ ╭──────╮  
││ │ Save ╰─────────────────────────────────────────────────────────────────╮
││ │    (syntax Save (save)                                                 │
││ │     (save path) (((Module) (Path) → Save-Failed _) Context path)       │
││ │     (save module path) (((Module) (Path) → Save-Failed _) module path) │
││ ├┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┤
││ │ Save will accept a path list that points to a location where the       │
││ │ Module, given as the first parameter, will be stored as a file.        │
││ │                                                                        │
││ │ If the first parameter is Nil, then the current top-level will be      │
││ │ stored.                                                                │
││ │                                                                        │
││ │ If the target of the path list does not exist, the Session will        │
││ │ attempt to create directory structure denoted by the path list.        │
││ │                                                                        │
││ │ If the target of the path list exists, the Session will attempt to     │
││ │ overwrite the existing file or Module.                                 │
││ │                                                                        │
││ │ If there is any implementation dependent failure when the Session      │
││ │ attempts to save the Module, this Expression will call the             │
││ │ 'save-failed effect.                                                   │
││ ╰────────────────────────────────────────────────────────────────────────╯
││
││ ╭──────╮  
││ │ Read ╰─────────────────────────────────────────────────────────────────╮
││ │                (syntax Read (read)                                     │
││ │                 (read) ((type (_ → End Character) (_ → End Program)))) │
││ ├┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┤
││ │ Read will consume characters from the Session duplex input stream and  │
││ │ interpret them as one of several forms of data.                        │
││ │                                                                        │
││ │ Depending on the desired result type, Read will either yield a Program │
││ │ or a single character.                                                 │
││ │                                                                        │
││ │ If the duplex input ends before Read can interpret a completed written │
││ │ Program or consume the next Character value, Read will call the 'eof   │
││ │ effect.                                                                │
││ ╰────────────────────────────────────────────────────────────────────────╯
││ 
││ ╭──────╮
││ │ Peek ╰─────────────────────────────────────────────────────────────────╮
││ │                             (syntax Peek (peek)                        │
││ │                              (peek) (((Natural) → End Character) 1)    │
││ │                              (peek n) (((Natural) → End Character) n)) │
││ ├┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┤
││ │ Peek will yield a Character value from the Session duplex input        │
││ │ without consuming it.                                                  │
││ │                                                                        │
││ │ Peek will accept a Natural number which denotes the number of          │
││ │ index positions in the duplex input beyond the current next Character  │
││ │ value which Peek will yield.                                           │
││ │                                                                        │
││ │ If the duplex input ends before Peek can yield the requested Character │
││ │ value, Peek will call the eof effect.                                  │
││ ╰────────────────────────────────────────────────────────────────────────╯
││ 
││ ╭───────╮
││ │ Print ╰────────────────────────────────────────────────────────────────╮
││ │               (syntax Print (print)                                    │
││ │                (print data) (((List) → Print-Failure Judgement) data)) │
││ ├┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┤
││ │ Print accepts a list of Symbols, Strings, and Expressions. It will     │
││ │ collate each item into a single string containing their written forms. │
││ │                                                                        │
││ │ The written form of an item will be identical to the String yielded by │
││ │ applying Format to that item.                                          │
││ │                                                                        │
││ │ The written form of each item will be appended to the Print String in  │
││ │ the order of its index as an item in the Expression invoking Print.    │
││ │                                                                        │
││ │ Expressions are not converted to their written form, but instead are   │
││ │ evaluated and their result is converted to written form.               │
││ │                                                                        │
││ │ After collating the written form of all the items in its invocation    │
││ │ Expression, Print will send the String data through the Session duplex │
││ │ output.                                                                │
││ │                                                                        │
││ │ If any implementation defined error occurrs when attempting to Print,  │
││ │ the 'print-failure effect is called.                                   │
││ ╰────────────────────────────────────────────────────────────────────────╯
││
││ ╭──────────╮
││ │ Evaluate ╰─────────────────────────────────────────────────────────────╮
││ │                             (syntax Evaluate (evaluate)                │
││ │                              (evaluate p) (((Program) → Invalid *) p)) │
││ ├┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┤
││ │ Evaluate accepts a Program and optionally a Module. It will evaluate   │
││ │ the Program, using the Module as the top-level if provided, as if the  │
││ │ Program was recieved from the Session input.                           │
││ │                                                                        │
││ │ If the Program provided to Evaluate cannot be validated, Evaluate will │
││ │ call the 'invalid Effect.                                              │
││ ╰────────────────────────────────────────────────────────────────────────╯

Enter : 
(syntax Enter (enter)
 (enter module) (((Module) → _) module))

Enter will set the Top-Level to the provided Module.

Leave : 
(syntax Leave (leave) (leave) ((_ → Quit _))
'(leave)
(_ → _)

Leave will set the Top-Level to the Module which was the previous Top-Level.
If the Top-Level is the Context, the Session shall terminate.

Quit :
(syntax Quit (quit) (quit) ((_ → Quit _)))
Quit will cause the Session to terminate.




││
│├ A path list, as used in Load and Save, is a list of Symbols and String data 
││ which corresponds to a directory structure. Each item in the list shall 
││ correspond to the name of a directory, with the last item in the list 
││ corresponding to the name of a file. String data shall correspond directly to 
││ a name in the filesystem which matches its Character contents. A Symbol shall
││ correspond to a name in the filesystem which matches the Character contents
││ of that Symbol's written form as if obtained by applying Format to that 
││ Symbol.

│                             ╭─────────────────╮
│                             ╽ Character Hooks │
│╭──────────────────────────────────────────────╯
│├ When the Session is reading from the duplex input, or Read has been invoked 
││ by a program, the Session employs a system of "hooks" in order to interpret
││ the Characters in the stream as NIL entities.
││ A "hook" is a pair of a Character value (a Unicode® code point) and a 
││ function which will perform the actual consumption of Character values and 
││ construct a NIL entity accordingly.
││
│├ A Session utilizes Hooks in the following manner...
││ 1. A Session will await Character values to be submitted to the duplex input.
││ 2. When a Character value is submitted, the Session will check the Hook 
││    Characters for a value that matches.
││ 3. If a Hook Character matches the Character from the duplex input, the 
││    associated Hook Function will be called.
││    If no Character matches the input Character, the Session will discard that
││    Character.
││ 4. After discarding the Character, or the Hook Function returning, the 
││    Session will repeat this process from step 1.
││
│├ A Hook Function will be invoked when the Session's Read capability encounters
││ a Character value in the duplex input stream which matches the Character value
││ for which that Hook is set.
││ 
│├ Hooks are global to the current live Session.
││
│├ Hooks are created or existing Hooks are updated with the following Macro...
││
││ ╭──────╮
││ │ Hook ╰─────────────────────────────────────────────────────────────────╮
││ │                                            '(hook ''string ''function) │
││ │          (((array Character 1) ((type (_ → *) _)) → (type Function _)) │
││ ├┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┤
││ │ Hook will accept a Character value given as a String of cardinality 1  │
││ │ and a Function which takes no arguments and may return any NIL entity. │
││ │                                                                        │
││ │ If the Function parameter is Nil, then Hook will yield the Function,   │
││ │ if one exists, otherwise Hook will yield Nil.                          │
││ ╰────────────────────────────────────────────────────────────────────────╯
││ 
││                             ╭───────────────╮
││                             ╽ Default Hooks │
││╭────────────────────────────────────────────╯
││├ When a Session starts, there are certain Hooks which are guaranteed to be 
│││ present before the Session begins reading from the duplex input, these Hooks
│││ are...
│││ ⎡ 
│││ ⎢ · "("
│││ ⎢ · "\""
│││ ⎢ · "'"
│││ ⎣
│││ These three hooks correspond to the three Character syntaxes which make up
│││ the written form of Programs. The Functions associated with these may be 
│││ implementation defined, but must conform to the formal grammar set out in 
│││ the subsection on Syntax in the following Chapter.
││╰╼
│╰╼
╰╼
                                  ╭──────────╮
                                  ╽ Programs │
╭────────────────────────────────────────────╯
├ Programs are the medium through which the behavior and effects of execution
│ can be specified. Programs are also a form of Data.
│
├ Programs are represented by a structure composed of Symbols and Expressions.
│ Symbols are atomic and cannot be decomposed into smaller structures.
│ Expressions are composite and contain sequences of other Symbols and
│ Expressions.
│
├ A "written Program" refers to the format used for storage and transmission of
│ NIL programs. Written Programs are represented by UTF-8 encoded Unicode® 
│ character sequences.
│
├ Syntax is used to draw a correspondence between a Program and its written form.
│
│                                  ╭────────╮
│                                  ╽ Syntax │
│╭──────────────────────────────────────────╯ 
│├ Syntax is used to establish a correspondence between the structure of a 
││ Program and its written form.
││
│├ The syntax of written programs is comprised of three rules...
││    
││ 1. A Symbol is any unbroken sequence of Unicode® glyphs excluding the  
││    character codes defined in the Character Category, "Z", and the ASCII round 
││    braces, "(" and ")". For berevity, the term "Blank" will be used to refer
││    to the set of characters in the Character Category, "Z".
││
││ 2. An Expression is denoted as a closed pair of round braces, "(" and ")" 
││    with an arbitrary number of characters between them.
││
││ 3. Symbols and Expressions which are items within another expression, must

│├ A written Program is considered malformed if the character sequence does not
││ contain evenly matched pairs of round braces. In all cases, every opening 
││ round brace must be followed by a closing round brace. Conversely, every 
││ closing round brace must be preceded by an opening round brace.
││
│├ The Unicode® character encoding also includes character codes which are
││ unprintable or otherwise invisible in a character sequence. These are 
││ specified in the Character Category, "C". Characters in the "C" category have
││ no meaning when included in a written program and must be ignored.
