Implementation Notes

What capabilities does my implementation have?

Necessary:
        Interpreter, Debugger, and Compiler 

Obvious:
        FFI, Filesystem Access, Threads

Good to have:
        IPC, Signals, POSIX-ish environment

The potential problem with this is fixing a set of capabilities and then becoming
outdated.
That's not technically a real problem however because of two facts.
Constrained environments don't have these capabilities and never will.
Future tech may have newer capabilities, but implementations can be extended.

Here's a tricky bit with my language.
Say I have a TUI implementation of a Session and I want to change my environment
to be a GUI implementation using a different module.
How would I do this?
I'd have to swap out the Session module and replace it with something else.
That's weird though.
Wouldn't I just configure the startup entry to hang up on the TUI and run the 
GUI?

I'm thinking about this wrong.
The main interface for a NIL program is the Session.
If you want a NIL program that runs in the terminal, you have to connect a 
terminal to it.
If you want a NIL program that runs in the GUI, you have to connect it to a GUI.
That's the point is that your program doesn't do anything until you make it do 
something.

Terminal apps have a specific interface.
Compiling a system app means you have to specify certain entry and exit functions
as well as cleanup and whatnot.
